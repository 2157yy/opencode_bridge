# Test Spec: OpenCode 多 CLI 智能体桥

## 1. 测试目标
验证桥是否满足：
- 主 CLI 为默认入口
- 子 CLI 为独立进程 / 独立终端窗口
- 可直接点名任意子 CLI
- 桥只负责路由与监督，不强依赖消息原文展示
- 可见监督维度为状态 / 阶段 / 产物

## 2. 测试范围
### In Scope
- 启动 / 注册 / 路由 / 状态汇总
- 主 CLI 与子 CLI 的独立性
- 产物索引与阶段变更
- 异常退出与恢复

### Out of Scope
- 完整消息原文回放
- 非 CLI 交互形态
- 具体 OpenCode 源码实现细节

## 3. 测试分层
### Unit
1. registry 记录 agent_id / role / window_id / session_id
2. 状态机从 queued -> running -> blocked -> produced -> done/failed 的转移
3. 产物索引可按 agent 查询
4. 路由器能将命令定向到指定 agent

### Integration
1. 桥后端可启动主 CLI
2. 桥后端可启动多个子 CLI 窗口
3. 子 CLI 注册后能被桥识别
4. 主 CLI 与子 CLI 可分别接收任务
5. 状态变更能同步到监督面板

### E2E
1. 用户从主 CLI 进入系统
2. 用户新增 2 个子 CLI
3. 用户直接给某个子 CLI 分配任务
4. 观察到该子 CLI 的状态、阶段、产物变化
5. 某个子 CLI 异常退出后可被标记并恢复

### Manual / UX
1. 终端窗口是否清晰区分主 / 子
2. 是否能从汇总面板快速定位到指定 agent
3. 是否能只看状态 / 阶段 / 产物就理解当前进展

## 4. 关键测试用例
### T1 主入口
- **Given** 桥启动完成
- **When** 人类进入系统
- **Then** 默认落到主 CLI

### T2 子 CLI 独立进程
- **Given** 桥请求启动 3 个子 agent
- **When** 它们全部启动
- **Then** 系统中存在 3 个独立 CLI 进程 / 终端窗口

### T3 直接点名子 CLI
- **Given** 某个子 agent 已注册
- **When** 人类直接向该 agent 发起任务
- **Then** 该任务只落到目标 agent

### T4 状态监督
- **Given** 子 agent 正在运行
- **When** 它完成一个阶段
- **Then** 桥可显示新阶段与产物摘要

### T5 不暴露消息原文也能监督
- **Given** 监督面板未显示原始消息
- **When** 用户查看 agent 详情
- **Then** 仍可通过状态 / 阶段 / 产物理解工作进展

### T6 故障与恢复
- **Given** 子 agent 异常退出
- **When** 桥检测到退出
- **Then** 状态标记为 failed，且可重连或重启

## 5. 验收标准
1. 主 CLI 与子 CLI 都是独立进程 / 窗口。
2. 路由支持默认到主 CLI 和显式到任意子 CLI。
3. 监督面板至少能展示状态、阶段、产物三类信息。
4. 消息原文不是监督的必要条件。
5. 异常退出可以被检测并恢复。

## 6. 回归风险点
- agent registry 丢失同步
- 状态机漏转移
- 路由误投递到错误 agent
- 终端窗口与 agent_id 映射错乱
- 退出恢复后产物索引不一致

## 7. 验证顺序
1. 先验单测覆盖状态机与 registry。
2. 再做集成测试覆盖启动 / 注册 / 路由。
3. 再做 E2E 覆盖主入口、子 CLI、监督与恢复。
4. 最后人工检查终端表现与汇总可读性。
